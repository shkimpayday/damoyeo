Technical Highlights
기술적 도전과 해결
1
로그인 유지 중 동시 요청 오류 해결
상황 — 로그인 후 발급되는 인증 토큰은 보안상 10분마다 만료됩니다. 문제는 토큰이 만료된 순간 페이지가 여러 데이터를 한꺼번에 불러오려 할 때 발생했습니다. 모든 요청이 동시에 "인증 실패"를 받으면서 토큰 갱신 요청이 중복으로 쏟아졌고, 각 갱신 요청이 서로 다른 새 토큰을 발급받아 쿠키에 덮어쓰면서 토큰 상태가 충돌했습니다. 결과적으로 일부 요청이 엇갈린 토큰으로 재시도되어 사용자가 갑자기 로그아웃되는 버그가 발생했습니다.

해결 — "첫 번째 요청만 실제 갱신을 진행하고, 나머지 요청은 줄 세워 기다리게" 하는 대기열 구조를 적용했습니다. 갱신이 완료되면 새 토큰으로 대기 중이던 요청을 한 번에 처리합니다.
let isRefreshing = false;
let failedQueue: { resolve: Fn; reject: Fn }[] = [];

// 첫 번째 인증 실패만 갱신 진행, 나머지는 대기
if (!isRefreshing) {
  isRefreshing = true;
  const tokens = await refreshToken();
  processQueue(tokens); // 대기 중인 요청 일괄 재실행
} else {
  return new Promise((resolve, reject) =>
    failedQueue.push({ resolve, reject })
  );
}

결과 — 토큰 갱신이 단 1번만 일어나고, 기다리던 요청이 모두 정상 처리됩니다. 의도치 않은 로그아웃이 완전히 사라졌습니다.
2
실시간 채팅에도 로그인 확인 적용하기
상황 — 일반 API 요청은 매번 로그인 여부를 확인할 수 있지만, 채팅에 사용하는 실시간 연결(WebSocket)은 처음 연결할 때 한 번만 확인이 가능하고 이후엔 변경이 불가합니다. 기존 로그인 검사 로직이 실시간 연결에는 아예 적용되지 않아 로그인 없이 채팅 채널에 접근할 수 있는 보안 구멍이 있었습니다.

해결 — 채팅 연결 시작 시점에 로그인 토큰을 함께 전달하고, 서버에서 이를 검증하는 별도 검사 단계를 추가했습니다.
// 채팅 연결 시 로그인 토큰을 함께 전달
const client = new Client({
  connectHeaders: {
    Authorization: `Bearer ${accessToken}`
  }
});

// 서버: 연결 시작 시점에 토큰 검증 후 사용자 확인

결과 — 모임 채팅, 정모 채팅, 고객지원 채팅 3곳 모두 동일한 방식으로 로그인한 사용자만 접근 가능하게 됐습니다.
3
고객상담 진행 상태 관리
상황 — 일반 채팅은 메시지만 주고받으면 되지만, 고객상담은 "누가 담당하고 있는지", "해결됐는지"를 추적해야 합니다. 이 구분 없이는 담당자가 겹치거나 답변 없이 묻히는 상담이 생길 수 있었습니다.

해결 — 상담 건마다 진행 단계를 명확히 구분했습니다.
대기 중 → 상담 진행 중 → 상담 완료
관리자 화면에서 단계별로 상담 목록을 필터링할 수 있고, 버튼 하나로 다음 단계로 전환합니다. 고객이 문의를 남기면 "대기 중"으로 시작하고, 관리자가 담당자를 지정하면 자동으로 "진행 중"으로 바뀝니다.
4
코드 수정하면 서버에 자동 반영
목표 — 코드를 고친 뒤 직접 서버에 접속해서 업데이트하는 번거로운 과정을 없애자.

코드 업로드 → 자동 빌드 → 이미지 저장소 전송 → 서버 자동 접속 → 서비스 재시작
코드를 GitHub에 올리는 것만으로 빌드부터 서버 배포까지 전 과정이 자동으로 처리됩니다. 서버 비밀번호나 접속 정보는 GitHub 보안 저장소에만 보관해 코드에 노출되지 않습니다. HTTPS 인증서 적용, SPA 라우팅, API 중계까지 자동화 파이프라인 안에서 함께 처리됩니다.
5
카카오페이 결제 연동
상황 — 외부 결제창은 팝업으로 열렸다 닫히는 방식이라, 결제가 끝난 뒤 그 결과를 우리 서버로 안전하게 전달받는 과정이 필요했습니다. 결제 도중 사용자가 창을 닫거나 뒤로 가는 경우도 모두 처리해야 했습니다.

구현
결제 준비 요청 → 카카오페이 결제창 → 결제 완료 코드 수신 → 최종 승인 처리
결제 취소·실패 시 각각 다른 안내 화면으로 이동하며, 승인 완료 후 프리미엄 구독 상태와 만료일이 자동으로 업데이트됩니다. 프리미엄 구독 결제와 유료 정모 참석 결제, 두 가지 흐름을 모두 지원합니다.
6
내 주변 모임 찾기
상황 — 키워드나 카테고리 검색만으로는 "지금 내 위치에서 가까운 모임"을 찾기 어렵습니다. 거리 계산을 서버에서 처리하지 않으면 수천 개의 모임 데이터를 모두 가져온 뒤 필터링해야 해서 성능 문제가 생깁니다.

해결 — 브라우저에서 현재 위치(위도·경도)를 받아 서버로 전달하면, 데이터베이스가 직접 거리를 계산해 가까운 순서로 결과를 돌려줍니다. 키워드·카테고리·거리 조건을 조합한 복합 검색도 하나의 쿼리로 처리됩니다. 한 번 허용한 위치 정보는 세션에 저장해 페이지 이동 시 불필요한 재요청을 방지합니다.
